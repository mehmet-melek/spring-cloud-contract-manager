Contract Testing Approaches

Consumer-Driven Contract (CDC)

Consumers define the contract that providers must fulfill

How It Works

1
Consumer Defines Contract

Consumer specifies expected API behavior and responses based on their needs

2
Provider Implements

Provider implements the API to satisfy the consumer's contract

3
Consumer Tests with Stubs

Consumer uses generated stubs from the contract for integration testing

Best Practices
  • • Maintain a centralized contract repository
  • • Automate contract verification in CI/CD
  • • Define clear contract ownership
Advantages
  • ✓ Consumer needs drive API design
  • ✓ Clear documentation of actual usage
  • ✓ Early breaking change detection
Challenges
  • • Complex N-N relationships
  • • GitOps complications
  • • Requires strong coordination

Provider-Driven Contract (PDC)

Provider writes the contract and generates stubs for consumers

How It Works

1
Provider Writes Contract

Provider directly authors the contract (OpenAPI or similar) for their API.

2
Unit Test Generation from Contract

Instead of writing controller unit tests, provider uses the contract to auto-generate unit tests and stubs.

3
Consumer Uses Provider's Stub Jar

Consumer adds the provider's generated stub jar as a dependency and uses it in their tests instead of custom mocks.

4
Contract Change & Build Feedback

If the contract changes and the consumer's build uses an outdated or incompatible stub jar, the consumer receives errors during build/test.

Best Practices
  • • Automate stub jar generation and publishing in CI/CD
  • • Document contract changes and notify consumers
  • • Encourage consumers to always use the latest stub jar
When to Use
  • • Provider wants full control over contract definition
  • • Teams prefer contract-driven unit test generation
  • • Consumers can adapt to provider's contract and use stubs
Advantages
  • ✓ Provider's process is similar to writing unit tests
  • ✓ Stub jar can be reused by multiple consumers
  • ✓ Automated feedback for contract changes
Challenges
  • • Coverage is low; business logic is not validated for consumers
  • • Consumers are limited to what provider exposes
  • • Relation/discovery is not guaranteed; stub jar usage is not enforced

OpenAPI-Driven Contracts

Leverage OpenAPI specifications for contract-first API development

How It Works

1
Create OpenAPI Document

Design your API contract using OpenAPI specification

2
Enhance with x-contract

Add test data and validation rules using x-contract extensions

3
Generate Test Assets

Automatically create mocks, stubs, and test cases

4
Verify Implementation

Run tests against your API implementation

Best Practices
  • • Maintain comprehensive OpenAPI documentation
  • • Automate contract generation in CI/CD
  • • Use semantic versioning for specs
When to Use
  • • API-first development approach
  • • When you have comprehensive OpenAPI specs
  • • For automated contract generation
Advantages
  • ✓ API-First approach promotion
  • ✓ Automated stub generation
  • ✓ Consistent documentation
Challenges
  • • Manual x-contract field addition
  • • No business logic testing
  • • Risk of over-engineering

OpenAPI Diff Analysis Approach

OpenAPI Diff Analysis

Proactive change detection using OpenAPI specification comparison

How It Works

1
Centralized OpenAPI Management

Store and version OpenAPI specifications in a centralized repository

2
Consumer Registration

Register consumers and their API dependencies

3
Automatic Diff Analysis

Automatically compare API versions and detect breaking changes

4
Proactive Notifications

Notify affected consumers about potential breaking changes

Best Practices
  • • Maintain high-quality OpenAPI specs
  • • Set up automated diff analysis
  • • Establish clear notification workflows
When to Use
  • • Large-scale microservices architecture
  • • When you need proactive change detection
  • • For regression prevention
Advantages
  • ✓ Zero development effort
  • ✓ High automation
  • ✓ Proactive detection
Challenges
  • • No business logic testing
  • • Depends on spec quality
